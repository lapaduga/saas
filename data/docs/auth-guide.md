# Авторизация и безопасность CloudApp

## Методы аутентификации

CloudApp поддерживает несколько методов аутентификации для обеспечения безопасности доступа к платформе.

### Email + Пароль

Базовый метод аутентификации. При входе используется email аккаунта и пароль.

**Требования к паролю:**

- Минимум 8 символов
- Хотя бы одна заглавная буква (A-Z)
- Хотя бы одна строчная буква (a-z)
- Хотя бы одна цифра (0-9)
- Рекомендуется: спецсимволы (!@#$%^&*)

**Блокировка аккаунта:**

После 5 неудачных попыток входа аккаунт блокируется на 15 минут. При дальнейших попытках блокировка увеличивается: 30 минут, 1 час, 24 часа. Для немедленной разблокировки напишите в поддержку.

### Двухфакторная аутентификация (2FA)

Дополнительный уровень безопасности, требующий код из приложения-аутентификатора.

**Настройка 2FA:**

1. Перейдите в **Настройки → Безопасность → 2FA**
2. Нажмите «Включить 2FA»
3. Отсканируйте QR-код в приложении (Google Authenticator, Authy, Microsoft Authenticator)
4. Введите 6-значный код из приложения для подтверждения
5. Сохраните резервные коды (10 штук) в безопасном месте

**Резервные коды:**

Каждый резервный код можно использовать однократно для входа, если основное устройство недоступно. Храните их в менеджере паролей или распечатайте.

**Отключение 2FA:**

Для отключения 2FA необходим текущий код аутентификатора или резервный код. Без этих данных отключение невозможно даже администратору.

### Single Sign-On (SSO)

Единый вход через корпоративный провайдер идентификации. Доступен на тарифах Pro и Enterprise.

**Поддерживаемые провайдеры:**

- **Okta** — SAML 2.0, OIDC
- **Azure Active Directory** — SAML 2.0, OIDC
- **Google Workspace** — SAML 2.0, OIDC
- **Custom SAML 2.0** — любой совместимый провайдер

**Настройка SSO через Okta:**

1. В консоли Okta создайте новое приложение (Applications → Create App Integration → SAML 2.0)
2. Укажите:
   - **Single sign-on URL:** `https://api.cloudapp.com/auth/sso/callback`
   - **Audience URI (SP Entity ID):** `https://api.cloudapp.com`
   - **Name ID format:** EmailAddress
3. Скопируйте метаданные (Identity Provider metadata)
4. В CloudApp перейдите в **Настройки → SSO → Добавить провайдер**
5. Вставьте метаданные и сохраните

**Маппинг атрибутов:**

| Атрибут Okta | Поле CloudApp |
|---|---|
| email | email |
| firstName | name.first |
| lastName | name.last |
| department | company |
| groups | role |

**Единый выход (SLO):**

При настройке SSO включите SLO для автоматического завершения сессии при выходе из корпоративного приложения.

## Управление сессиями

### Токены доступа

CloudApp использует JWT-токены для аутентификации API-запросов:

- **Access Token** — действует 1 час, используется для API-запросов
- **Refresh Token** — действует 30 дней, используется для получения нового Access Token

**Обновление токена:**

```bash
POST /api/v1/auth/refresh
{
  "refresh_token": "rt_abc123..."
}
```

**Отзыв токенов:**

Все активные сессии можно отозвать в **Настройки → Безопасность → Сессии**. Это полезно при подозрении на компрометацию аккаунта.

### Журнал входов

В разделе **Настройки → Безопасность → Журнал входов** отображаются:

- Дата и время последних 50 входов
- IP-адрес и геолокация
- Устройство и браузер
- Метод аутентификации
- Статус (успешно / неудачно)

## API-авторизация

### Типы API-ключей

**Personal API Key:**

- Привязан к конкретному пользователю
- Имеет права этого пользователя
- Подходит для личного использования и тестирования

**Service Account Key:**

- Привязан к сервисному аккаунту
- Может иметь расширенные права
- Подходит для автоматизации и интеграций

**OAuth 2.0:**

- Поток Authorization Code для веб-приложений
- PKCE для SPA и мобильных приложений
- Client Credentials для серверных приложений

### Использование API-ключа

```bash
# Через заголовок Authorization
curl -H "Authorization: Bearer sk_live_abc123..." https://api.cloudapp.com/v1/files

# Через query parameter (не рекомендуется)
curl "https://api.cloudapp.com/v1/files?api_key=sk_live_abc123..."
```

### Области доступа (Scopes)

| Scope | Описание |
|---|---|
| `files:read` | Чтение файлов и папок |
| `files:write` | Создание, загрузка, удаление файлов |
| `files:share` | Управление правами доступа |
| `users:read` | Просмотр пользователей |
| `users:manage` | Управление пользователями |
| `webhooks:manage` | Управление вебхуками |
| `analytics:read` | Просмотр аналитики |

## Шифрование данных

### Шифрование при хранении (At Rest)

Все данные шифруются алгоритмом **AES-256-GCM**. Ключи шифрования хранятся в отдельном HSM (Hardware Security Module) и автоматически ротируются каждые 90 дней.

### Шифрование при передаче (In Transit)

Все соединения используют **TLS 1.3**. Поддерживается HSTS (HTTP Strict Transport Security). SSL-сертификаты обновляются автоматически.

### Шифрование пользовательских ключей

Для Enterprise-клиентов доступна опция **Client-Side Encryption (CSE)**:

- Данные шифруются на стороне клиента перед загрузкой
- Ключи шифрования никогда не покидают клиентское устройство
- CloudApp не имеет доступа к содержимому файлов

## Соответствие стандартам

### SOC 2 Type II

CloudApp соответствует требованиям SOC 2 Type II по критериям:

- Безопасность (Security)
- Доступность (Availability)
- Целостность обработки (Processing Integrity)
- Конфиденциальность (Confidentiality)
- Приватность (Privacy)

### GDPR

Платформа полностью соответствует Общему регламенту по защите данных ЕС:

- Право на доступ (Article 15)
- Право на удаление (Article 17)
- Право на перенос данных (Article 20)
- Ограничение обработки (Article 18)
- Уведомление о нарушениях (Articles 33-34)

### ISO 27001

Система менеджмента информационной безопасности (ISMS) сертифицирована по ISO 27001. Сертификат обновляется ежегодно.

## Рекомендации по безопасности

1. Включите 2FA для всех пользователей
2. Используйте SSO для команд от 10 человек
3. Регулярно проверяйте журнал входов
4. Ротируйте API-ключи каждые 90 дней
5. Настройте оповещения о подозрительной активности
6. Используйте Service Account Key вместо Personal для автоматизации
7. Настройте IP-whitelist для Enterprise-аккаунтов
